## 1. 🏗️ System Architecture & Tech Achievement

### 🔹 트리거(Trigger) 기반 데이터 무결성 제어
- **스터디 구성원 동기화 (trg_member_withdraw_sync):** 회원이 탈퇴할 때 데이터를 즉시 삭제하지 않고 상태(status)만 변경하는 시스템 특성상, 기존의 ON DELETE CASCADE를 사용할 수 없었습니다. 이를 해결하기 위해 회원 상태가 변경될 때 해당 회원의 스터디 가입 정보를 자동으로 삭제(DELETE FROM study_member)하는 트리거를 구현하여 데이터 무결성을 완벽히 보장했습니다.
- **정원 초과 방지 (trg_check_study_capacity):** 스터디 가입 시 최대 정원(capacity)과 현재 가입 인원을 비교하여 정원 초과 가입을 원천 차단하는 복합 트리거를 설계 및 적용했습니다. 이 과정에서 데이터 증가에 따른 부하를 인지하고, 향후 성능 최적화를 위한 비정규화(현재 인원수 컬럼 추가) 아이디어를 도출하기도 했습니다.

### 🔹 N:N 관계 해소 및 복합 제약조건을 통한 데이터 신뢰성 확보
- **관계 테이블 설계:** 회원(Member)과 스터디(Study) 간의 다대다(N:N) 관계를 데이터베이스 상에서 논리적으로 구현하기 위해 `study_member` (스터디 구성원) 관계 테이블을 도출하여 가입자 관리를 구조화했습니다.
- **Unique 제약조건(UQ) 적용:** 시스템 상 한 회원이 동일한 스터디에 중복으로 가입하는 치명적인 오류를 방지하기 위해, 관계 테이블 내에 '회원번호 + 스터디번호' 조합으로 고유(Unique) 제약조건을 설정했습니다.
- **참조 무결성 유지 (ON DELETE CASCADE):** 부모 데이터인 스터디가 삭제될 경우, 연결된 자식 데이터(스터디 일정, 할 일 등)가 DB에 무의미하게 남는 고아 데이터(Orphan Data) 현상을 방지하고자 `ON DELETE CASCADE`를 적용하여 데이터의 정합성을 엄격히 유지했습니다.

### 🔹 Servlet/JSP (Model 2) 계층 분리와 방어적 프로그래밍
- **SRP (단일 책임 원칙) 적용:** 요청 제어(Controller), 비즈니스 로직(Service), 데이터 접근(DAO) 계층을 엄격히 분리하여 결합도를 낮췄습니다. 
- **데이터 정합성 검증:** 스터디 정보 수정 시 현재 가입된 인원 수(totalCountM)보다 정원을 낮게 설정할 수 없도록 Java 백엔드 단에서 강력한 예외 처리 방어벽을 구축했습니다.

## 2. 📝 Retrospective: 협업과 풀스택 개발의 기반 다지기

이번 프로젝트는 단순한 기능 구현을 넘어, 팀원들과 시스템을 연결하고, 프론트엔드부터 백엔드까지 전체 흐름을 주도적으로 파악하는 중요한 변곡점이 되었습니다.

- **JSP & Servlet 흐름 숙달:** Servlet이 Java 영역과 Tomcat 웹 컨테이너 사이에서 어떻게 동작하는지, 그리고 JSP 페이지와 파라미터(Parameter)를 어떻게 주고받으며 활용해야 하는지 그 내부적인 동작 원리를 확실하게 체득할 수 있었습니다.
- **사전 설계와 네이밍 컨벤션의 중요성:** 개발 초기, 팀원 간 Method 및 Parameter 명명 규칙을 명확하게 합의하지 않고 시작하여 디버깅과 코드 통합에 상당한 어려움을 겪었습니다. 남이 짠 코드를 분석하고 유지보수하는 과정을 직접 겪으며, 프로젝트 시작 전 꼼꼼한 아키텍처 설계와 통일된 컨벤션의 필요성을 뼈저리게 느꼈습니다.
- **프론트엔드(UI) 개발의 재미 발견:** 백엔드 로직에 치중하다 보니 CSS나 UI 작업이 높게만 느껴졌습니다. 하지만 Tailwind CSS, Daisy UI, 그리고 JavaScript를 활용해 화면을 구성해 보며, 반복 숙달을 거치면 UI 영역 또한 논리적으로 통제 가능하고 꽤 큰 재미를 느낄 수 있는 분야라는 것을 깨달았습니다.
- **Git을 활용한 협업 능력 향상:** GitHub Repository와 Branch를 적극 활용하여 병렬 작업을 진행했습니다. 형상 관리 도구를 사용하는 것이 처음에는 낯설고 충돌(Conflict) 해결에 애를 먹기도 했지만, 기능을 익히고 나니 Git 없이는 다수의 개발자가 참여하는 현대적인 애플리케이션 개발이 불가능하다는 확신을 얻었습니다.
- **Oracle Database 활용 능력의 진화:** 이전 기초 프로젝트에서는 단순한 DML 작성이 주를 이루었으나, 이번 프로젝트에서는 복잡한 제약조건(제한된 정원, 연관 테이블 참조 등)을 위반하지 않으면서도 단시간 내에 논리적으로 완벽한 대량의 더미 데이터를 생성해야 했습니다. 이를 위해 고급 Oracle 문법을 스스로 찾아 연구하고 활용하면서, 데이터베이스를 다루는 시야와 활용 능력이 한층 깊어졌습니다.
