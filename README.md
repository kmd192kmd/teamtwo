## 1. 🏗️ System Architecture & Tech Achievement

### 🔹 트리거(Trigger) 기반 데이터 무결성 제어
- **스터디 구성원 동기화 (trg_member_withdraw_sync):** 회원이 탈퇴할 때 데이터를 즉시 삭제하지 않고 상태(status)만 변경하는 시스템 특성상, 기존의 ON DELETE CASCADE를 사용할 수 없었습니다. 이를 해결하기 위해 회원 상태가 변경될 때 해당 회원의 스터디 가입 정보를 자동으로 삭제(DELETE FROM study_member)하는 트리거를 구현하여 데이터 무결성을 완벽히 보장했습니다.
- **정원 초과 방지 (trg_check_study_capacity):** 스터디 가입 시 최대 정원(capacity)과 현재 가입 인원을 비교하여 정원 초과 가입을 원천 차단하는 복합 트리거를 설계 및 적용했습니다. 이 과정에서 데이터 증가에 따른 부하를 인지하고, 향후 성능 최적화를 위한 비정규화(현재 인원수 컬럼 추가) 아이디어를 도출하기도 했습니다.

### 🔹 비동기 통신과 프론트엔드 연동을 통한 UX 최적화
- **Fetch API 도입:** 세부 할 일(To-Do)의 완료 상태를 변경할 때 새로고침 없이 Fetch API를 활용하여 비동기로 백엔드와 통신(todo-status.do)하도록 설계했습니다. 
- **즉각적인 시각적 피드백:** 비동기 요청 후 응답받은 JSON 결과에 따라 JavaScript로 즉각적인 DOM 조작(취소선 및 배경색 처리)을 수행하여 사용자 경험(UX)을 크게 향상시켰습니다.
- **모던 UI 및 시각화:** Tailwind CSS와 DaisyUI를 활용해 일관된 디자인 시스템을 구축하고, FullCalendar 라이브러리를 연동하여 스터디 일정을 직관적인 달력 형태로 제공했습니다.

### 🔹 Servlet/JSP (Model 2) 계층 분리와 방어적 프로그래밍
- **SRP (단일 책임 원칙) 적용:** 요청 제어(Controller), 비즈니스 로직(Service), 데이터 접근(DAO) 계층을 엄격히 분리하여 결합도를 낮췄습니다. 
- **데이터 정합성 검증:** 스터디 정보 수정 시 현재 가입된 인원 수(totalCountM)보다 정원을 낮게 설정할 수 없도록 Java 백엔드 단에서 강력한 예외 처리 방어벽을 구축했습니다.

## 2. 🤖 Retrospective: AI 시대, 개발자의 경쟁력

### "질문의 깊이가 설계의 수준을 결정한다"
프로젝트 전반에 AI를 적극 활용하며, "코드를 생성해주는 AI 시대에 인간 개발자의 역할은 무엇인가?"에 대해 깊이 고민했습니다. AI는 구현 시간을 획기적으로 단축해 주었지만, 전체적인 시스템의 뼈대를 세우고 예외 상황을 시뮬레이션하는 것은 결국 **개발자의 집요함**이었습니다.

- **설계에 대한 집요함:** ERD 설계 단계에서 단순히 테이블을 나누는 것에 그치지 않고, "미래에 발생할 수 있는 극단적인 트래픽 상황"을 가정하며 팀원들과 치열하게 토론할 시간을 AI 덕분에 확보할 수 있었습니다.
- **문제 해결의 주체성:** AI는 답을 제안할 뿐, 그 답의 트레이드오프(Trade-off)를 인지하고 비판적으로 수용하여 우리 시스템 아키텍처에 부합하는지 최종 결정하는 것은 개발자의 몫입니다. AI가 짠 코드를 복사-붙여넣기만 한다면 장기적인 성장은 불가능합니다.
- **결론:** AI 시대에도 여전히 학습은 필요하며, AI의 의견을 무비판적으로 수용하지 않고 효율적으로 학습해 나가는 것이 필수 역량임을 느꼈습니다. 저의 경쟁력은 AI가 대체할 수 없는 **'본질을 파악하는 질문 능력'**과 **'최적의 구조를 설계하기 위해 끝까지 파고드는 집요함'**에 있다고 확신합니다.

단순히 코드를 작성하는 사람을 넘어, 시스템의 철학을 정립하고 기술적 한계를 돌파하는 **주도적인 아키텍트**로 성장해 나갈 것입니다.
