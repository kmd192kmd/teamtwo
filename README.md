## 1. 🏗️ System Architecture & Tech Achievement

### 🔹 트리거(Trigger) 기반 데이터 무결성 제어 및 성능 개선 고찰
- **스터디 구성원 동기화 (trg_member_withdraw_sync):** 회원이 탈퇴할 때 데이터를 즉시 삭제하지 않고 상태(status)만 변경하는 시스템 특성상, 기존의 ON DELETE CASCADE를 사용할 수 없었습니다. 이를 해결하기 위해 회원 상태가 변경될 때 해당 회원의 스터디 가입 정보를 자동으로 삭제(DELETE FROM study_member)하는 트리거를 구현하여 데이터 무결성을 완벽히 보장했습니다.
- **정원 초과 방지 및 한계점 개선 (trg_check_study_capacity):** 스터디 가입 시 최대 정원(capacity)과 현재 가입 인원을 비교하여 정원 초과 가입을 원천 차단하는 복합 트리거를 설계 및 적용했습니다. 하지만 이 과정에서 데이터가 많아질수록 복합 트리거 연산으로 인한 DB 부하가 증가하는 문제를 인지하게 되었습니다. 이를 통해 향후에는 **비정규화를 적용하여 스터디 테이블에 '현재 인원수' 컬럼을 추가하고, 가입/탈퇴 시 단순 `UPDATE` 트리거와 `CHECK` 제약조건(현재 인원수 <= 최대 정원)만으로 검증 로직을 구성하는 것이 성능 최적화에 훨씬 유리하다는 아키텍처 개선점**을 깨달았습니다.

### 🔹 N:N 관계 해소 및 복합 제약조건을 통한 데이터 신뢰성 확보
- **관계 테이블 설계:** 회원(Member)과 스터디(Study) 간의 다대다(N:N) 관계를 데이터베이스 상에서 논리적으로 구현하기 위해 `study_member` (스터디 구성원) 관계 테이블을 도출하여 가입자 관리를 구조화했습니다.
- **Unique 제약조건(UQ) 적용:** 시스템 상 한 회원이 동일한 스터디에 중복으로 가입하는 치명적인 오류를 방지하기 위해, 관계 테이블 내에 '회원번호 + 스터디번호' 조합으로 고유(Unique) 제약조건을 설정했습니다.
- **참조 무결성 유지 (ON DELETE CASCADE):** 부모 데이터인 스터디가 삭제될 경우, 연결된 자식 데이터(스터디 일정, 할 일 등)가 DB에 무의미하게 남는 고아 데이터(Orphan Data) 현상을 방지하고자 `ON DELETE CASCADE`를 적용하여 데이터의 정합성을 엄격히 유지했습니다.

### 🔹 Servlet/JSP (Model 2) 계층 분리와 방어적 프로그래밍
- **SRP (단일 책임 원칙) 적용:** 요청 제어(Controller), 비즈니스 로직(Service), 데이터 접근(DAO) 계층을 엄격히 분리하여 결합도를 낮췄습니다.

## 2. 📝 후기 (김민도)

- **JSP, Servlet 숙달**
이번 프로젝트를 통해 servlet이 java와 tomcat 사이에서 어떻게 동작하는지, jsp페이지와는 parameter값을 어떻게 주고받으며 그 값들을 어떻게 활용해야 하는지 숙달할 수 있었습니다.

- **사전 설계의 중요성**
method, parameter 명명법을 확실하게 정해두고 시작하지 않아서 디버깅에 어려움이 있었고, 다른 사람이 작성한 코드를 유지보수해야 하는 경우 많은 어려움이 따를 것으로 예상됩니다. 협업 시에는 method, parameter 명명법 또한 팀원들과 통일해야 할 필요성을 느꼈습니다.

- **UI 개발 경험 (tailwind + daisy ui + javascript)**
CSS, UI 개발 경험이 부족하여 벽처럼 느껴졌었는데, 이번 프로젝트를 통해 경험함으로써 CSS, UI 또한 반복 숙달하여 학습한다면 재미를 느끼며 개발할 수 있는 영역이라는 것을 느꼈습니다.

- **협업 능력 향상 (git)**
git-hub 협업 repository, branch를 사용하여 병렬작업을 경험함으로써 협업 능력이 향상되었습니다. 새로운 것을 배우는 것이 늘 그렇듯 처음에는 힘들었지만, 사용법을 익히고 나니 git을 사용하지 않고서는 협업 개발이 불가능하다는 생각까지 들었습니다.

- **Oracle 이해도 향상**
지난 DB 프로젝트는 첫 번째 프로젝트이다 보니 요구사항 분석, 테이블 설계, 각종 문서 정리 등 기본적인 문서 작업들에 익숙해지는 과정이었고 더미 데이터 생성은 insert문을 이용한 작업이었습니다. 이번 프로젝트에서는 풀스택 개발 과정이어서 더미 데이터 생성 비중이 높지 않았는데, 오히려 그 덕분에 oracle 고급 문법들을 찾아보고 활용하여 제약 조건을 위반하지 않는 데이터를 생성할 수 있었고, 벽처럼 느껴지던 고급 문법 사용 경험을 가질 수 있었습니다.
